# TIL: 공유메모리 + 락프리 링큐로 프로세스간 통신

## 왜 링큐인가

연결리스트로 락프리 큐를 만들면 노드 동적할당, 포인터 관리, ABA 문제까지 따라온다. 반면 링큐는 처음에 메모리를 한 번만 잡고, 인덱스만 움직이면 된다. 공유메모리 환경에서는 특히 차이가 크다. 연결리스트는 프로세스마다 mmap 주소가 달라서 포인터를 전부 오프셋으로 바꿔야 하지만, 링큐는 인덱스 기반이라 그런 게 없다.

## SPSC 링큐 구조

```
생산자 → tail만 건드림
소비자 → head만 건드림
서로 상대방 값은 읽기만 함
```

CAS 없이 `acquire/release`만으로 충분하다. tail+1 == head면 full, head == tail이면 empty. 슬롯 하나를 항상 비워두므로 실제 용량은 N-1.

```c
bool push(RingQueue* q, Item item) {
    uint32_t tail = atomic_load_explicit(&q->tail, memory_order_relaxed);
    uint32_t next = (tail + 1) & (N - 1);

    if (next == atomic_load_explicit(&q->head, memory_order_acquire))
        return false; // full

    q->slots[tail] = item;
    atomic_store_explicit(&q->tail, next, memory_order_release);
    return true;
}

bool pop(RingQueue* q, Item* item) {
    uint32_t head = atomic_load_explicit(&q->head, memory_order_relaxed);

    if (head == atomic_load_explicit(&q->tail, memory_order_acquire))
        return false; // empty

    *item = q->slots[head];
    atomic_store_explicit(&q->head, (head + 1) & (N - 1), memory_order_release);
    return true;
}
```

## 공유메모리 올리기

`shm_open` + `mmap` 조합. `/dev/shm`은 tmpfs라 메모리 기반이라 빠르고, 프로세스가 죽어도 파일이 남는다.

```c
// 생산자
int fd = shm_open("/my_queue", O_CREAT | O_RDWR, 0666);
ftruncate(fd, sizeof(RingQueue));
RingQueue* q = mmap(NULL, sizeof(RingQueue), PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);
close(fd);

// 소비자
int fd = shm_open("/my_queue", O_RDWR, 0666);
RingQueue* q = mmap(NULL, sizeof(RingQueue), PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);
close(fd);

// 정리
munmap(q, sizeof(RingQueue));
shm_unlink("/my_queue");
```

C에서는 `memset(q, 0, sizeof(RingQueue))`으로 초기화하면 끝. C++에서는 placement new로 생성자를 강제 호출해야 한다.

## 비지 폴링 없애기: Adaptive Waiting

락프리인데 CPU를 낭비하지 않으려면 **adaptive waiting** 패턴을 쓴다.

```
spin (PAUSE 명령) → futex sleep
```

멀티 프로세스 환경에서는 `sched_yield`가 의미없다. 같은 프로세스 내 스레드에게만 양보하는 거라 다른 프로세스 생산자한테는 효과가 없다. 바로 futex로 가면 된다.

```c
bool pop_adaptive(RingQueue* q, Item* item) {
    int spin = 0;

    while (true) {
        if (try_pop(q, item)) return true;

        if (spin < SPIN_COUNT) {
            __builtin_ia32_pause(); // x86 PAUSE
            spin++;
        } else {
            uint32_t val = q->futex_val.load(memory_order_relaxed);
            if (try_pop(q, item)) return true; // 자기 전 마지막 확인

            syscall(SYS_futex, &q->futex_val, FUTEX_WAIT, val, NULL, NULL, 0);
            spin = 0;
        }
    }
}
```

## futex 동작 원리

futex는 공유메모리의 int 하나 + 커널 대기 큐다. 커널이 그 **물리 주소**를 키로 대기 큐를 관리하기 때문에 프로세스가 달라도 같은 공유메모리를 가리키면 동작한다.

`FUTEX_WAIT`에 val을 넘기는 게 핵심이다. 커널이 현재 값을 확인해서 val과 같을 때만 재운다. 생산자가 이미 push하고 값을 바꿨으면 즉시 리턴한다. 이게 race condition을 막는 전부다.

```c
// 프로세스 간에는 PRIVATE 붙이면 안됨
syscall(SYS_futex, &q->futex_val, FUTEX_WAIT, val, NULL, NULL, 0); // O
syscall(SYS_futex, &q->futex_val, FUTEX_WAIT_PRIVATE, val, NULL, NULL, 0); // X
```

## false sharing 주의

head와 tail이 같은 캐시라인에 있으면 생산자/소비자가 서로 캐시를 무효화한다. 64바이트 패딩 필수.

```c
typedef struct {
    atomic_uint head;
    char _pad1[60];
    atomic_uint tail;
    char _pad2[60];
    atomic_uint futex_val;
    char _pad3[60];
    Item slots[N];
} RingQueue;
```

## 정리

- 공유메모리 IPC에는 링큐가 연결리스트보다 훨씬 단순하고 빠르다
- SPSC는 CAS 없이 acquire/release만으로 충분
- 비지 폴링 제거는 spin → futex 순서로, 멀티 프로세스면 sched_yield 단계는 빼도 된다
- futex는 물리 주소 기준으로 동작해서 프로세스 간에도 쓸 수 있다
- false sharing 방지를 위해 캐시라인 패딩은 항상 챙길 것
