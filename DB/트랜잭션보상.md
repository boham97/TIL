# 대용량 INSERT 청크 커밋 시 보상 트랜잭션 패턴

## 문제 상황

대용량 INSERT를 단일 트랜잭션으로 처리하면 DB에 과부하가 걸리므로, 보통 N건씩 **청크 단위로 나눠서 커밋**한다.
그런데 청크 3에서 실패하면, 이미 커밋된 청크 1·2는 **DB 트랜잭션 롤백이 불가능**하다.

→ 비즈니스 관점의 "논리적 롤백"이 필요하며, 이를 **보상 트랜잭션(Compensating Transaction)** 이라고 한다.

---

## 핵심 개념

보상 트랜잭션은 이미 커밋된 작업의 효과를 **역으로 되돌리는 별도의 트랜잭션**이다.
DB의 ROLLBACK이 아니라, 애플리케이션 레벨에서 직접 "되돌리기 로직"을 구현하는 것이다.

---

## 해결 전략 3가지

### 1. 배치 ID + 삭제 (가장 단순)

모든 row에 `batch_id`를 심어두고, 실패 시 해당 ID로 일괄 DELETE한다.

```java
String batchId = UUID.randomUUID().toString();

for (List<Item> chunk : partition(items, chunkSize)) {
    try {
        insertChunk(chunk, batchId);  // 각 row에 batch_id 포함
    } catch (Exception e) {
        compensate(batchId);  // DELETE FROM items WHERE batch_id = ?
        throw e;
    }
}
```

- 장점: 구현이 간단하다
- 단점: DELETE 비용 발생, 중간에 다른 서비스가 데이터를 읽을 수 있다

### 2. 상태 플래그 (삭제 대신 무효화)

INSERT 시 `status = 'PENDING'`으로 넣고, 전체 성공 시 `CONFIRMED`, 실패 시 `CANCELLED`로 변경한다.

```java
String batchId = UUID.randomUUID().toString();

try {
    for (List<Item> chunk : partition(items, chunkSize)) {
        insertChunkAsPending(chunk, batchId);  // status = 'PENDING'
    }
    confirmBatch(batchId);   // UPDATE SET status = 'CONFIRMED'
} catch (Exception e) {
    cancelBatch(batchId);    // UPDATE SET status = 'CANCELLED'
    throw e;
}
```

- 장점: 물리 삭제 없이 감사 추적(audit trail) 가능
- 단점: 조회 쿼리에 `WHERE status = 'CONFIRMED'` 조건이 항상 필요하다
- 주의: PENDING 데이터가 다른 서비스에 노출되지 않도록 조회 조건을 강제해야 한다

### 3. 스테이징 테이블 (가장 안전)

임시 테이블에 청크를 쌓고, 전부 성공하면 본 테이블로 한 번에 이동한다.

```java
createStagingTable();

try {
    for (List<Item> chunk : partition(items, chunkSize)) {
        insertToStaging(chunk);
    }
    migrateToMainTable();  // INSERT INTO main SELECT * FROM staging
} catch (Exception e) {
    truncateStaging();     // TRUNCATE = 거의 무비용
    throw e;
} finally {
    dropStagingTable();
}
```

- 장점: 본 테이블이 오염되지 않는다 (중간 상태 데이터 노출 없음)
- 단점: 스테이징 → 본 테이블 이동 시 단일 트랜잭션으로 처리해야 하므로 데이터가 크면 부담

---

## 비교 요약

| 기준 | 배치 ID + 삭제 | 상태 플래그 | 스테이징 테이블 |
|---|---|---|---|
| 구현 난이도 | 낮음 | 중간 | 중간 |
| 보상 비용 | DELETE 비용 | UPDATE만 | TRUNCATE (거의 무비용) |
| 본 테이블 오염 | 잠깐 노출됨 | PENDING으로 노출 | 전혀 없음 |
| 감사 추적 | 불가 | 가능 | 별도 구현 필요 |

---

## 실무 판단 기준

- 다른 서비스가 해당 테이블을 **실시간 조회**하고 있다면 → **스테이징 테이블**
- **감사 추적**이 중요한 도메인이라면 → **상태 플래그**
- 단순 배치 작업이라면 → **배치 ID + 삭제**

---

## 추가로 기억할 것

- 보상 트랜잭션 자체도 실패할 수 있다. 반드시 **재시도 로직 + 실패 로그**를 남겨야 한다.
- 보상 트랜잭션 패턴은 **Saga 패턴**의 핵심 구성 요소이기도 하다 (MSA 환경에서의 분산 트랜잭션).
- 청크 사이즈는 보통 1,000~10,000건 사이에서 DB 부하와 트랜잭션 로그를 보며 튜닝한다.
