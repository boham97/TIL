# TIL: UDP가 빠르긴 해요?

**2026-06-23**

"UDP = 무조건 빠름"은 흔한 오해. 정확히는 **"커널이 해주던 일을 안 하니까 그만큼 가볍다"**가 본질. "빠르다"를 **지연(latency)**으로 보면 대체로 맞고, **처리량(throughput)**으로 보면 자주 틀린다.

## send 한 번 자체는 안 빠르다

TCP든 UDP든 결국 같은 `sendto`/`recvfrom` 시스템콜을 탄다. 호출 1회의 커널 진입 비용은 거의 같다. "UDP는 함수 호출이 빨라서 빠르다"는 틀린 그림.

## UDP가 진짜 빠른 지점 — 지연(latency)

TCP가 하는 일을 안 하기 때문에 첫 바이트가 빨리 간다.

- **핸드셰이크 없음** — TCP는 3-way handshake로 RTT 한 번 까먹음. UDP는 바로 쏨. (DNS가 UDP인 이유)
- **ACK 안 기다림** — fire-and-forget.
- **Head-of-line blocking 없음** — TCP는 패킷 하나 유실되면 그 뒤 도착분까지 막힘. UDP는 도착한 건 바로 줌. (게임·음성·영상에서 결정적)
- **Nagle 알고리즘 없음** — TCP는 작은 패킷을 모으려 일부러 지연(`TCP_NODELAY`로 끔). UDP는 그런 버퍼링 없음.

→ 작고, 시간 민감하고, 가끔 유실돼도 괜찮은 트래픽에서 빠름. 게임·VoIP·실시간 영상·DNS.

## UDP가 안 빠르거나 오히려 느린 지점 — 처리량

큰 데이터 대량 전송은 현대 TCP가 보통 더 빠르다. 수십 년치 최적화가 박혀 있음.

- **세그멘테이션 오프로드(TSO/GSO/GRO)** — 큰 덩어리를 한 번에 넘기면 알아서 잘라줌. 순진한 UDP는 패킷마다 `sendto` → 시스템콜 폭증.
- **sendfile zero-copy**, **window scaling**, **BBR/CUBIC 혼잡 제어**로 링크를 꽉 채워 씀.
- 순진한 UDP 대량 전송은 혼잡 제어가 없어 → 막 쏘면 유실 폭주, 조심하면 링크 놀림. 둘 다 손해.

## 가장 큰 함정: 신뢰성 다시 붙이면 도로아미타불

UDP가 빠른 건 신뢰성·순서·흐름제어를 떼어냈기 때문. 그걸 UDP 위에 직접(ACK·재전송·순서번호·혼잡제어) 구현하면:
- 결국 TCP가 하던 일을 다시 하게 되고,
- 커널의 고도 최적화보다 유저스페이스에서 더 잘 짜기 어렵고,
- "UDP라서 빨랐던" 이점이 상당 부분 사라진다.

이걸 제대로 해낸 게 **QUIC**(HTTP/3) — UDP 위에 TCP급 신뢰성을 얹으면서 핸드셰이크·HOL blocking을 줄임. 단, 막대한 엔지니어링의 결과지 "UDP 쓰면 공짜로 빨라짐"이 아님.

## 정리

| 기준 | UDP가… |
|---|---|
| 첫 바이트까지 지연 | **빠름** (핸드셰이크·ACK 없음) |
| 패킷 유실 시 뒤 패킷 | **안 막힘** (HOL blocking 없음) |
| 연결당 메모리/상태 | **가벼움** (소켓 하나로 다수 처리) |
| 대량 전송 처리량 | TCP가 **보통 더 빠름** (오프로드·혼잡제어) |
| 신뢰성 필요할 때 | 직접 구현 → **이점 상쇄** |

**한 줄 요약:** UDP는 "더 빠른 통신"이 아니라 **"일을 덜 하는 통신"**이다. 그 덜 하는 일이 마침 필요 없으면(가끔 유실 OK + 지연이 생명) 빠른 거고, 필요하면 결국 그 일을 떠안아야 해서 안 빠르다.
